[![R-CMD-check](https://github.com/k-shibuki/bridle/actions/workflows/R-CMD-check.yaml/badge.svg)](https://github.com/k-shibuki/bridle/actions/workflows/R-CMD-check.yaml)

AI knowledge harness generation framework for R statistical packages

---

## Statement of Need

When LLMs drive statistical analysis, the entire decision chain becomes probabilistic. Choosing effect measures, specifying models, assessing heterogeneity — as these decisions cascade, small uncertainties at each step accumulate, making it impossible to guarantee methodological correctness. LLMs are generally accurate at local judgments ("A or B in this situation?") but unreliable at designing the overall flow ("which decisions, in what order?").

Moreover, statistical analysis requires **pre-specification (protocols)**. In meta-analysis, the analytical approach must be fixed in advance with a clear understanding of models, protocols, and the direction of analysis (what is confirmatory vs. exploratory). R packages such as meta and metafor offer rich options, but an LLM cannot make appropriate choices unless it understands the "meaning" and "applicability conditions" of those options.

bridle generates **harnesses** that structure R package knowledge and deliver it to the LLM. The LLM presents considerations and suggestions based on this knowledge base. The user makes the final call. We use the horse's (LLM's) power, but the rider (harness) sets the direction. The name bridle derives from this control structure.

## Architecture

bridle has two functions: a **plugin generation pipeline** (a build tool that semi-automatically generates knowledge plugins from R packages) and a **runtime engine** (an execution environment that controls the dialogue between the LLM and the user based on plugin knowledge).

The **plugin generation pipeline** analyzes an R package's `formals()`, Rd files, and source code, then combines the results with literature information to semi-automatically generate a knowledge plugin.

A **plugin** is a self-contained knowledge base for a single R package. It includes a decision graph that defines the order of decisions, knowledge entries for each decision point, and technical constraints. Most content is LLM-drafted + expert-reviewed.

The **runtime engine** follows the decision graph to control the conversation flow, supplying relevant knowledge to the LLM at each decision point.

## Plugin Design

Plugins are defined **per R package**. For example, `bridle.meta` is the knowledge base for the meta package, and `bridle.metafor` is the knowledge base for the metafor package; each functions independently. A plugin holds both methodological knowledge and API mapping in a self-contained manner.

### Decision Graph

The decision graph defines a flow that efficiently narrows the parameter space. Each node corresponds to a decision point (e.g., choosing an effect measure or a tau-squared estimator), and transition conditions determine the next decision point ([ADR-0002](docs/adr/0002-decision-graph-flow-control.md)).

```yaml
# inst/bridle/decision_graph.yaml (meta::metabin example, excerpt)
graph:
  entry_node: sm_selection
  nodes:
    sm_selection:
      type: decision
      topic: effect_measures
      parameter: sm
      transitions:
        - to: method_selection
          always: true

    method_selection:
      type: decision
      topic: pooling_methods
      parameter: method
      transitions:
        - to: continuity_correction
          when: "zero cells exist"
          computable_hint: "any(event.e == 0 | event.c == 0)"
        - to: model_type
          otherwise: true

    heterogeneity_assessment:
      type: diagnosis
      topic: heterogeneity
      transitions:
        - to: sensitivity_analysis
          when: "heterogeneity is high"
          computable_hint: "I2 > 50"
        - to: publication_bias
          otherwise: true

    sensitivity_analysis:
      type: decision
      topic: sensitivity_approaches
      transitions:
        - to: execute_analysis   # loop: change parameters and re-run
          when: "re-run with different parameters"
        - to: publication_bias
          otherwise: true
```

Four node types exist: `context_gathering` (collecting context), `decision` (parameter selection), `execution` (code generation and execution), and `diagnosis` (post-fit diagnostics). Loops can occur via transitions from `diagnosis` nodes (e.g., sensitivity analysis with different parameters).

### Knowledge Format

Knowledge is **descriptive**: instead of "recommend PM," it states "PM has these properties, supported by these references." The LLM generates context-dependent recommendations from this. Knowledge entries link to decision graph nodes via the `topic` field.

```yaml
# inst/bridle/knowledge/tau2_estimators.yaml
topic: tau2_estimators
target_parameter: method.tau
package: meta
function: metabin

entries:
  - id: tau2_small_k
    when: "number of studies is small (fewer than 5)"
    computable_hint: "k < 5"
    properties:
      - "PM is robust with small samples"
      - "REML is prone to convergence issues"
      - "DL tends to underestimate"
    references:
      - "Veroniki AA, et al. (2016) Res Synth Methods"
      - "IntHout J, et al. (2014) BMC Med Res Methodol"

  - id: tau2_moderate_k
    when: "number of studies is moderate (5 to 30)"
    computable_hint: "k >= 5 & k <= 30"
    properties:
      - "REML offers a good balance of bias and efficiency"
    related:
      - "consider Hartung-Knapp adjustment (method.random.ci)"
    references:
      - "Langan D, et al. (2019) Stat Med"
```

`when` is a natural-language text string that serves as the expert-review target. `computable_hint` is an optional R expression that the runtime evaluates programmatically when possible; if evaluation fails, the LLM judges from the `when` text ([ADR-0003](docs/adr/0003-when-condition-semantics.md)).

### Technical Constraints

`constraints/technical.yaml` is auto-generated by `scan_package()`. It holds inter-parameter technical constraints extracted from formals default expressions and source code.

```yaml
# inst/bridle/constraints/technical.yaml (auto-generated)
constraints:
  - id: peto_requires_or
    source: formals_default
    type: forces
    condition: "method == 'Peto'"
    forces: {sm: "OR"}

  - id: valid_sm_values
    source: rd_description
    type: valid_values
    param: sm
    values: ["RR", "OR", "RD", "ASD", "DOR", "VE"]
```

## Plugin Generation Pipeline

The pipeline, provided by bridle core, generates most of a plugin's content.

### 1. `bridle::scan_package("meta")`

Analyzes the R package's parameter space in three layers ([ADR-0004](docs/adr/0004-scanner-three-layer-analysis.md)).

**Layer 1 — formals() analysis**: Parses default expressions to extract inter-parameter dependency graphs. Because R formals retain default values as unevaluated expressions, conditional defaults like `sm = ifelse(!is.na(charmatch(tolower(method), ...)), "OR", ...)` yield a `method -> sm` dependency mechanically. Automatic parameter classification (data input / statistical decision / presentation, etc.) also happens here.

**Layer 2 — Rd analysis**: Extracts valid value lists (e.g., "One of `"RR"`, `"OR"`, ..."), parameter descriptions, and reference lists from man pages.

**Layer 3 — Source code analysis**: Confirms valid value enumerations from `match.arg()` calls and extracts runtime constraints from `stop()` / `warning()` conditions.

The output is a `ScanResult` object containing the parameter list, dependency graph, valid-value map, technical constraints, and reference list.

### 2. `bridle::fetch_references(scan_result)`

Resolves DOIs and retrieves abstracts via CrossRef API / PubMed API. Fetches open-access full text where available.

### 3. `bridle::draft_knowledge(scan_result, references)`

Uses ellmer to draft the following:

- **decision_graph.yaml** — The LLM drafts an efficient decision ordering using the dependency graph and parameter classifications from the scan result. Note that the dependency direction in formals (e.g., `method -> sm`) and the decision direction (e.g., `sm -> method`) can be reversed.
- **knowledge/*.yaml** — Drafts descriptive knowledge entries from package documentation + literature.
- **constraints/technical.yaml** — Auto-generated from the scan result (no AI drafting needed).

### 4. Expert Review

Direct YAML editing: verifying decision graph ordering, adding competing views, refining when conditions, supplementing references.

### 5. `bridle::validate_plugin("bridle.meta")`

- Coverage check: are there parameters not covered by the decision graph?
- Consistency check: do knowledge topics match decision graph nodes?
- Constraint check: do constraint references point to real parameters?
- Reachability check: are all nodes reachable from the entry node?

Knowledge quality improves incrementally: formals() only -> abstracts added -> full text added -> expert review.

## Runtime Engine

The runtime engine controls the dialogue according to the plugin's decision graph.

### Graph Engine

Manages the current node in the decision graph and evaluates transition conditions to determine the next node. When a transition has a `computable_hint`, the engine attempts to evaluate it as an R expression; on success, it transitions accordingly. When evaluation is not possible (e.g., data not yet loaded), it delegates the judgment to the LLM.

Loops are supported: a transition from a `diagnosis` node back to an `execution` node allows re-running with changed parameters (e.g., sensitivity analysis, post-diagnostic refitting).

### Knowledge Retriever

Retrieves knowledge entries matching the current node's `topic` along with relevant constraints. Each topic typically has 5-15 entries, all injected into the LLM context in full. If the knowledge base scales up, selective retrieval via ragnar can be adopted, but this is unnecessary in the initial phase.

### Prompt Assembler

Combines retrieved knowledge entries, technical constraints, user context (data characteristics, decision history so far), and protocol information into a structured prompt. The prompt includes knowledge properties and references so the LLM can generate context-dependent recommendations grounded in descriptive knowledge.

### Response Parser

Interprets the LLM response, extracting user-facing content (considerations, suggestions, code) and graph transition signals.

## Protocol Layer

Statistical analysis has **analysis hierarchies**: confirmatory, sensitivity, exploratory, and post-hoc. The protocol layer provides abstractions in bridle core, with specifics defined in plugin templates.

Plugins provide protocol templates via `protocol_templates/*.yaml`. The LLM uses template metadata (required fields, guidance, nodes that must be pre-specified) to guide the user in a wizard-like fashion. This supports both protocol development and protocol-based analysis.

## Usage Example

Interactive dialogue example using `bridle.metafor` for a pairwise meta-analysis.

```r
library(bridle.metafor)

agent <- bridle_agent(plugins = "metafor", provider = "anthropic")
agent$console()
```

## Technology Stack

Built on the Posit ecosystem.

| Package | Role |
|---|---|
| [{ellmer}](https://ellmer.tidyverse.org/) | Unified LLM API interface. Supports Claude, OpenAI, Gemini, Ollama. Streaming, tool calling, structured data extraction |
| [{ragnar}](https://ragnar.tidyverse.org/) | RAG workflows. For selective knowledge retrieval when the knowledge base scales up. Not required in the initial phase since per-node full injection suffices |
| [{vitals}](https://vitals.tidyverse.org/) | LLM evaluation framework. Data-driven measurement of judgment accuracy |
| [{mcptools}](https://github.com/posit-dev/mcptools) | R implementation of MCP (Model Context Protocol). MCP server/client support |
| [{S7}](https://rconsortium.github.io/S7/) | Next-generation class system promoted by R Consortium |
| [{cli}](https://cli.r-lib.org/) | Terminal UI |

Reference retrieval uses CrossRef API and PubMed Entrez API.

## Interfaces

Two interfaces, delivered incrementally.

**Interface A — R CLI REPL**
A custom REPL loop based on `readline()`. The harness holds the initiative for flow control. Highest control fidelity; implemented first.

**Interface B — MCP Server**
Exposes bridle as an MCP server via mcptools, allowing LLM clients (Cursor, Claude Code, etc.) to call it as a tool. The LLM uses harness functions as tools. Added after Interface A stabilizes.

## Development Environment

Development uses a containerized R environment. No local R installation is required.

```bash
make container-build   # Build container (rocker/tidyverse + renv)
make container-up      # Start container
make renv-init         # Initialize renv (first time only)
make doctor            # Verify environment
```

R package dependencies are managed by [renv](https://rstudio.github.io/renv/) (`renv.lock` is the single source of truth). RStudio Server is available at `http://localhost:8787` when the container is running.

Run `make help` for all available targets.
